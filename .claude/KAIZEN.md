# hyper-chinchiro 改善点と開発計画

現状レビュー（2026-07-05時点）で洗い出した課題と、優先度順の開発計画。

---

## 🔴 P0: 問題（早めに対処すべき）

### 1. デバッグキーがpublicリポジトリに露出している
- リポジトリが public のため、`DEBUG_KEY` とデバッグページのURLパスが誰でも閲覧可能
- 悪意ある人が任意のルームの出目・能力を操作できる状態
- **対策**: キーを環境変数化（Workers の secret + Next.js の env）、またはデバッグメッセージを本番で無効化するフラグ

### 2. リロード・回線切断で即脱落する
- サーバーは `onClose` で即 `removePlayer` するため、スマホのスリープ復帰・リロードで自分が消えてスコアも失われる
- 実プレイで最も痛い問題（モバイルは頻繁に接続が切れる）
- **対策**: 切断後の猶予期間（例: 60秒）を設け、同じ接続ID（partysocketの `_pk` は再接続で維持される）で復帰したらそのまま継続

### 3. 全員切断でゲームが完全消滅する
- 状態がDurable Objectのメモリにしかないため、全員が一瞬切れるとゲームが初期化される
- **対策**: `ctx.storage`（SQLite）への状態保存。partyserver では `onStart` で復元する形

### 4. サーバー未接続時のフィードバックがない
- WebSocketが繋がらないと「接続中…」のまま無言（ローカル開発でwranglerを起動し忘れると詰まる、今日実際に発生）
- **対策**: 数秒でタイムアウト表示「サーバー({HOST})に接続できません」＋再試行ボタン

## 🟡 P1: プレイ体験の改善（UI観点）

### 5. 残り振り直し回数が見えない
- `rollCountMap` はサーバーにあるがUI未表示。「あと何回振れるのか」が分からない
- **対策**: ActionBarの「振る」ボタンに「振る（あと2回）」のように表示

### 6. ラウンド履歴・ゲームログがない
- 過去ラウンドで誰が何を出して何pt動いたかを後から確認できない
- 検証コンソールには精算表示を付けたが、ゲーム本体にも欲しい
- **対策**: サーバーでラウンド履歴を蓄積し、折りたたみ式のログUIを追加

### 7. ゲーム終了時の演出がない
- game_over時に順位表彰・最終結果画面がなく、淡々と終わる
- **対策**: 最終スコア順位表示（1位を大きく）＋紙吹雪などの演出

### 8. ルール・役の説明がゲーム内にない
- チンチロを知らない人は役の強さも倍率も分からない
- **対策**: ヘルプモーダル（役の一覧表・強さ順・倍率、能力一覧）

### 9. サイコロの見た目が数字テキスト
- `DiceDisplay` は数字を表示するだけ。アニメーションも0.45秒のwobbleのみ
- **対策**: ピップ（目のドット）描画、回転アニメーション強化、確定時の効果

### 10. ランダムモードで能力の発表演出がない
- 手番の能力がさりげなくテキストで変わるだけで気づきにくい
- **対策**: 手番開始時に「今回の能力: シゴロ賽！」のカード演出

### 11. 観戦状態の明示がない
- ゲーム進行中に入室した人はjoinできず観戦になるが「観戦中」の表示がない
- **対策**: 観戦バッジ＋「次のゲームから参加できます」の案内

## 🟢 P2: ゲームデザイン上の課題

### 12. シゴロ賽が強すぎる（選択固定モードで縮退）
- 4・5・6しか出ない = 役なし・ヒフミが原理的に出ず、最低でも目4、456とゾロ目も高確率
- ランダムモードでは rarityWeight 1 で調整済みだが、**選択固定モードでは全員シゴロ賽を選ぶのが最適解**になってしまう
- **対策案**（いずれか）:
  - 選択固定モードで同一能力の重複選択を禁止（早い者勝ち or ドラフト制）
  - 強能力に代償を付ける（シゴロ賽: 負けた時の支払い2倍、など）
  - 選択固定モードでは強能力（シゴロ賽・神の一手）を選択不可にする

### 13. ヒフミ賽が弱すぎる（ネタ枠）
- 1・2・3しか出ない = 最高でピンゾロ(3.7%)、大半は目1〜3かヒフミ
- ロマン枠として意図的なら良いが、報酬が釣り合っていない
- **対策案**: ピンゾロ時の倍率をさらに上げる、またはヒフミを出しても支払いが発生しない特典を付ける

### 14. 引き分け＝親勝ちの明示
- 同値の役は親の勝ちという仕様（一般的なチンチロでは「流れ（ノーカウント）」もある）
- 子に不利なことがUI上どこにも書かれていない
- **対策**: 仕様として維持するならルール説明に明記。または引き分けは移動なしに変更

### 15. ラウンド数が人数で不公平になる
- maxRounds固定5なので、3人プレイだと親を2回やる人と1回の人が出る（親は有利/不利が大きい）
- **対策**: maxRoundsを人数の倍数に自動調整（3人なら6ラウンド）

### 16. 持ち点・賭け金の概念がない
- 全精算が1pt基準の勝敗ポイント制で、リスク選択の駆け引きがない
- マイナス無制限なので大差がつくと消化試合になる
- **対策案**: 持ち点制（例: 10pt開始、0で脱落）、子が賭け金を選べる制度（1〜3pt）

### 17. 能力効果の体感が薄い
- 確率変更系は数ゲームでは効果が実感しにくい
- **対策**: 出目確定時に能力が影響した可能性の演出（例: 能力の目が出たらバッジ光る）、統計表示（このゲームでの出目分布）

## 🔵 P3: 技術基盤

### 18. 型定義の二重管理
- `party/src/types/game.ts` と `apps/web/src/types/game.ts` を手動で同期しており、片方だけ更新する事故が起きやすい
- **対策**: モノレポの共有パッケージ `packages/shared/` に切り出し

### 19. 結合テストがない
- stateMachineの単体テスト（39件）はあるが、フェーズ遷移を通したゲーム全体のシナリオテストがない
- **対策**: 「2人でjoin→ready→5ラウンド→game_over→再戦」のシナリオテストを vitest で追加

### 20. CI がない
- push時にテスト・型チェック・ビルドが自動実行されない
- **対策**: GitHub Actions で `tsc / vitest / next build` を回す

## 開発計画（フェーズ分け）

### Phase A: 安定性と安全性（P0対応）
1. DEBUG_KEY の環境変数化（Workers secret + `NEXT_PUBLIC_` なしのサーバー検証）
2. 切断猶予期間の実装（60秒のreconnect grace）
3. `ctx.storage` によるゲーム状態の永続化・復元
4. 接続失敗時のエラー表示と再試行

### Phase B: プレイ体験（P1のクイックウィン）
1. 残り振り直し回数の表示（小粒・効果大）
2. ゲーム終了時の順位表彰画面
3. ルール・役・能力のヘルプモーダル
4. ラウンド履歴ログ
5. サイコロのピップ表示とアニメーション強化

### Phase C: ゲームバランス（P2、プレイテストと並行）
1. 選択固定モードの能力重複禁止（または強能力のBAN）
2. ヒフミ賽・シゴロ賽の報酬/代償調整
3. maxRoundsの人数連動
4. （検討）持ち点制・賭け金制の導入

### Phase D: 基盤整備（P3、随時）
1. GitHub Actions CI
2. 共有型パッケージ化
3. ゲームシナリオ結合テスト
